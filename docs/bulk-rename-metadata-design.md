# Bulk Rename and Metadata Editing Design

**Status:** Draft Design  
**Date:** May 2026  
**Owner:** Find maintainers  
**Related:** Bulk operations roadmap, data integrity strategy

---

## Problem Statement

Bulk rename and metadata editing are useful features for managing large photo libraries, but they introduce significant data consistency risks:

- **Filename changes** can desynchronize actual object keys in MinIO, database metadata, and user expectations
- **Metadata edits** (captions, tags, likes, custom fields) are stored across three systems: PostgreSQL `media_json`, the vector search index, and MinIO object metadata
- **Partial failures** during bulk operations can leave individual images in inconsistent states (e.g., renamed in DB but not in MinIO)
- **Undo/retry complexity** increases dramatically without a clear transaction boundary
- **Concurrency collisions** arise if users edit the same image while a bulk operation is in flight

Find's core promise is **local-first privacy with data integrity**. A poor bulk-edit design could violate this by corrupting or losing user data.

---

## Goals & Non-Goals

### Goals
- Enable users to batch-rename files and edit metadata for multiple images at once
- Maintain data consistency across MinIO, PostgreSQL, and vector search at all times
- Define a clear transaction boundary so partial failures are recoverable
- Reduce operational complexity of future bulk actions (delete, tag, cluster feedback)
- Make the undo/retry behavior predictable and transparent to users

### Non-Goals
- Rename actual files on disk (Find only manages uploads to MinIO)
- Support renaming multiple images to the *same* filename (enforce uniqueness)
- Implement full ACID rollback across all three storage systems in this phase
- Provide collaborative editing (assume single-user local instance)
- Auto-generate filenames or intelligent batch naming (user-driven only)

## Safe Bulk Editable Fields

Below is the field-by-field analysis of what can be safely bulk-edited:

| Field | Storage | Editability | Rationale |
|-------|---------|-------------|-----------|
| **Display Filename** | `media.filename` (PostgreSQL) | ✅ **Safe** | Rename stored in DB only; MinIO key unchanged. No cascading references. |
| **Liked Status** | `media.liked` (PostgreSQL) | ✅ **Safe** | Boolean flag in DB only. No external dependencies. |
| **Caption** | `media.metadata_json.caption` (PostgreSQL) | ✅ **Safe** | Stored in `metadata_json` JSONB. Can be updated atomically per record. Vector index will be regenerated on next reprocessing if caption changes meaning. |
| **Tags / Custom Metadata** | `media.metadata_json.tags` (PostgreSQL) | ✅ **Safe** | Custom user tags in `metadata_json`. Isolated from ML-generated fields. |
| **Object Metadata (EXIF, Width, Height)** | `media.exif_json`, `media.width`, `media.height` | ⚠️ **Restricted** | Read-only; derived from image analysis. Editing breaks audit trail and image integrity. |
| **Auto-Generated Metadata (Caption, Objects, OCR)** | `media.metadata_json` (ML fields) | ⚠️ **Restricted** | Generated by ML pipeline. Editing user-corrected versions should go through feedback system, not bulk edit. |
| **File Hash** | `media.file_hash` (PostgreSQL) | ❌ **Unsafe** | Unique constraint and duplicate detection anchor. If changed, violates data model. |
| **MinIO Object Key** | `media.minio_key` (PostgreSQL) | ❌ **Unsafe** | Changes require object move in MinIO, DB update, search index update, and URL regeneration. Not atomic. Deferred to Phase 2. |
| **Cluster ID** | `media.cluster_id` (PostgreSQL) | ❌ **Unsafe** | Derived from clustering algorithm. Editing breaks clustering audit trail. Use clustering feedback endpoints instead. |
| **Vector Embedding** | `media.vector` (pgvector column) | ❌ **Unsafe** | Generated by ML pipeline. Editing defeats semantic search. Regenerate via reprocessing if caption changes. |

**Recommendation:** Initial bulk edit scope = **filename, liked status, caption, custom tags only**.

## Filename vs Object Key Decision

When a user bulk-renames images in the UI, there are three behavioral options:

### Option A: Rename Both Filename and MinIO Object Key (Full Rename)

**What happens:** User changes "photo_001.jpg" → "sunset_vacation.jpg"
- Update `media.filename` in PostgreSQL
- **Move object** in MinIO from `<original-key>` to `<new-key>` (copy + delete or native move API)
- Regenerate presigned URLs for the new key
- Update any cached metadata

**Pros:**
- Clean user experience; display name and actual object are in sync
- Filename and object key semantically aligned

**Cons:**
- Requires MinIO operation for every rename; slow for large batches (hundreds of renames)
- Move operation is not atomic: if DB updates but MinIO move fails, object is orphaned
- Partial failures leave inconsistent state (image shows new name but old key still in MinIO)
- Undo becomes complex: must reverse both DB and MinIO state
- Higher risk of race conditions if image is being downloaded during rename

---

### Option B: Rename Only the Display Filename (DB Only)

**What happens:** User changes "photo_001.jpg" → "sunset_vacation.jpg"
- Update `media.filename` in PostgreSQL only
- MinIO object key stays unchanged
- Display shows new name; underlying object key is opaque to user

**Pros:**
- Fast: single SQL UPDATE, no external I/O
- Atomic: rename always succeeds or fails cleanly
- Undo is trivial: restore old filename from audit log
- No risk of orphaned objects or MinIO inconsistency
- Partial failures affect only DB, no cascading damage
- Scales to thousands of renames in seconds

**Cons:**
- Filename and MinIO object key diverge
- If user exports/downloads images, file extension may not match actual content (minor cosmetic issue)
- Advanced users may expect object key to reflect filename (cultural expectation)

---

### Option C: Hybrid Approach (Configurable Per Batch)

**What happens:** Offer users a choice: "Rename display only" vs. "Rename files on disk" (power user option)

**Pros:**
- Flexibility for different use cases
- Power users can opt into full rename if willing to accept latency

**Cons:**
- Adds UI/API complexity and decision paralysis
- Still requires solving Option A's atomicity problem for the full-rename path
- Confusing for beginners: "Why does my file have two names?"

---

### Recommendation: **Option B (Display Filename Only)**

**Rationale:**
1. **Data safety first:** Single-system writes (PostgreSQL only) eliminate cascading failure modes.
2. **Undo simplicity:** Audit trail + filename reversion is straightforward.
3. **Scales:** Supports bulk rename of 1000+ images in <1 second without external bottlenecks.
4. **Partial failure recovery:** If a batch rename fails halfway, you know exactly which records failed (DB constraints) and can retry safely.
5. **Future migration path:** If we later decide to support MinIO renames, we can build a separate "consolidate object keys" maintenance job that runs in the background and syncs keys to filenames.

**User Communication:** Present bulk rename as "Update how images appear in your gallery" rather than "rename on disk." This sets correct expectations.

## Undo, Retry & Error Handling Strategy

### Undo Behavior (Product Level)

**Requirement:** All bulk edit operations must be reversible within a reasonable time window (e.g., 30 days or until session ends).

**Implementation approach:**
- Maintain an audit log table: `bulk_operation_audit(id, operation_type, media_ids, changes_before, changes_after, user_id, created_at, reversed_at)`
- On successful bulk edit, store the operation record with `changes_before` (old values) and `changes_after` (new values) as JSON
- Provide an "Undo last bulk operation" button in the UI
- Undo applies the reverse operation: copy `changes_before` back to affected records
- Mark operation as `reversed_at = now()` (don't delete; maintain audit trail)

**Limitations:**
- Undo is only available for the most recent operation per user session (or per day; configurable)
- If user edits a field individually after a bulk edit, undo will not revert that individual edit
- Undo is best-effort: if file no longer exists, undo silently skips that record

---

### Retry Behavior (Product Level)

**Requirement:** Failed bulk operations must be retryable with clear visibility into what failed.

**Implementation approach:**
- Bulk edit endpoint returns immediately with operation ID: `{ operation_id: "bulk-123", status: "pending" }`
- Backend processes edits asynchronously via RQ job (same pattern as image upload)
- Job updates status as it progresses: `pending` → `processing` → `partial_success` or `success` or `failed`
- Store per-record results in operation record: `{ media_id: 42, result: "success" } | { media_id: 43, result: "failed", error: "filename_duplicate" }`
- UI polls operation status endpoint to show progress: "Renaming 150 images... 142 done, 8 failed"
- User can retry failed records by ID or retry entire operation (skips already-succeeded records)

---

### Error Handling (Product Level)

| Error Type | Example | Handling |
|-----------|---------|----------|
| **Validation Error** | User provides duplicate target filenames | Reject entire batch before starting. Return error list to user. No partial execution. |
| **Constraint Violation** | Target filename already exists | Skip that record, log error, continue with others. Report in result summary. |
| **Database Error** | Concurrent edit conflict | Retry record up to 3 times. If still fails, mark as failed and continue. |
| **Missing Record** | Image deleted while bulk op in flight | Skip gracefully; report as skipped in summary. |
| **Timeout** | Job takes >30 minutes | Kill job, mark as partial_failure, allow user to retry remaining records. |

## Proposed UX Flow

### Bulk Rename Workflow

1. **Selection:** User selects 1–50 images in gallery using checkboxes or multi-select
2. **Action:** Clicks "Bulk Rename" button in toolbar
3. **Dialog:** Modal opens with table showing:
   - Current filename
   - Input field for new filename
   - Live validation: green checkmark if unique, red error if duplicate
   - "Auto-suffix" option: `{name}_batch` applied to all
4. **Confirm:** User reviews changes, clicks "Apply"
5. **Submission:** Frontend POSTs to `/api/bulk-edit` with `{ operation: "rename", media_ids: [...], updates: { filename: "..." } }`
6. **Feedback:** Backend returns `{ operation_id: "bulk-123", status: "pending" }`
7. **Progress:** UI shows spinner + progress bar; polls `/api/bulk-edit/{operation_id}` every 1–2 seconds
8. **Completion:** Shows result summary:
   - ✅ 48 renamed successfully
   - ⚠️ 2 failed (filename already exists)
   - "Undo" button appears for 30 minutes
   - "Retry failed" button if failures exist

### Bulk Metadata Edit Workflow

1. **Selection:** User selects images
2. **Action:** Clicks "Edit Metadata" button
3. **Dialog:** Multi-field form:
   - Checkbox: "Update caption" + text input
   - Checkbox: "Update tags" + tag input (multi-value)
   - Checkbox: "Toggle liked" (set all to liked/unliked)
4. **Confirm & Submit:** Same as rename (POST to `/api/bulk-edit` with different operation type)
5. **Progress & Completion:** Same flow as rename

## Technical Considerations

### Architecture

**Database:**
- All bulk operations on `media` table using SQLAlchemy batch updates
- Audit log stored in new `bulk_operation_audit` table (migration required)
- No changes to `media.minio_key` in this phase; filenames edited in `media.filename` only

**API Endpoint:**
```
POST /api/bulk-edit
{
  "operation": "rename" | "metadata",
  "media_ids": [1, 2, 3, ...],
  "updates": {
    "filename": "new_name.jpg",  // for rename
    "caption": "...",             // for metadata
    "liked": true,
    "tags": ["tag1", "tag2"]
  }
}
```

**Response:**
```json
{
  "operation_id": "bulk-op-uuid",
  "status": "pending",
  "job_id": "rq-job-id"
}
```

**Worker Job:**
- RQ worker processes bulk edit job asynchronously
- Job iterates over media_ids, validates each, applies updates
- Stores result per media_id: success or failure reason
- Updates operation record with final status and detailed results

---

### Concurrency & Safety

**Problem:** User selects 100 images, clicks Bulk Rename, then individually edits one of those images before the job completes.

**Solution:**
- Bulk edit job reads media record at job-start time, applies update if record still matches expected state
- Use SQLAlchemy's `version_id` or timestamp-based optimistic locking if critical
- If record changed since selection, mark that record as "skipped: changed since selection" in results
- Warn user if partial success detected

**Problem:** Two bulk operations target the same image ID simultaneously.

**Solution:**
- Queue operations serially per media_id or per operation (first-come-first-served)
- OR: Use database row-level locking during update: `SELECT ... FOR UPDATE`
- Document assumption: single-user local instance, so high contention is unlikely in MVP

---

### Search Index & Vector Embeddings

**Question:** When caption is bulk-edited, should the vector embedding be regenerated?

**Answer:** No, not in this phase.
- Embeddings are generated by ML pipeline during image indexing
- If user bulk-edits a caption, vector stays the same (based on original caption)
- This is acceptable because:
  - Search vector was pre-computed; updating it is expensive
  - User can trigger re-indexing of specific images if they want updated embeddings
  - Bulk edit is primarily for display/organization, not for ML re-analysis
- **Future enhancement:** Add "Re-embed selected images" as separate bulk action

---

### Data Migration & Backward Compatibility

- Add `bulk_operation_audit` table via Alembic migration
- Existing media records unaffected; nullable audit reference is OK
- No breaking API changes if endpoint versioning is used

### Risks & Mitigations

| Risk | Mitigation |
|------|-----------|
| Job queue overload if many users bulk-rename concurrently | Limit queue size, use priority queue, document single-user assumption |

| Duplicate filename check is racy (file added between check and update) | Use DB unique constraint as safety net; catch violation, retry with new name |

## Open Questions

1. **Undo retention policy:** Should undo be available indefinitely or expire after 30 days? Recommend: 30 days (balances recoverability vs. audit table size).

2. **Bulk edit batch size limit:** What's the UX threshold (10, 50, 100, 1000 images)? Recommend: Start with 100; scale up based on performance testing.

3. **Filename collision resolution:** If user renames Image A → "vacation.jpg" and Image B already has that name, should we auto-suffix one or reject the batch? Recommend: Reject entire batch; require user to disambiguate.

4. **Search index sync:** After bulk caption edit, should search results update immediately or require a reindex? Recommend: Results update immediately (old embeddings OK); document limitation in release notes.

5. **Audit log access:** Should users see the audit log via API or only through undo UI? Recommend: Undo UI only in MVP; audit API in Phase 2 for power users.

---

## Summary

This design provides a **safe, scalable foundation** for bulk editing by:
- Limiting initial scope to filename + simple metadata (avoiding MinIO complexity)
- Using asynchronous jobs + audit trails for clear transaction boundaries
- Defining per-record error handling so partial failures don't cascade
- Offering simple undo/retry workflows for users

The recommended Phase 1 implementation (bulk rename + undo) can be completed in 2–3 days and unlocks confidence for Phase 2 (metadata edits) and Phase 3 (MinIO renames).
